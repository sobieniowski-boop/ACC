# Taxonomy P1 — Brand/Category Gap Fill (2026-03-05)

## Scope
Wdrożono P1 taxonomy dla braków `brand/category/product_type` z:
- tabelami predykcji i review queue,
- auto-apply dla wysokiego confidence,
- runtime fallback w `Inventory` i `Profit`.

## DB schema
Serwis `app/services/taxonomy.py` utrzymuje:
- `dbo.acc_taxonomy_node`
- `dbo.acc_taxonomy_alias`
- `dbo.acc_taxonomy_prediction`

`acc_taxonomy_prediction` zawiera m.in.:
- klucze encji (`sku`, `asin`, `ean`),
- sugestie (`suggested_brand`, `suggested_category`, `suggested_product_type`),
- `confidence`, `source`, `status`,
- ślad review (`reviewed_by`, `reviewed_at`).

## Prediction pipeline
`refresh_taxonomy_predictions()`:
1. Pobiera kandydatów z brakami metadanych.
2. Buduje sugestie z 3 źródeł:
   - `pim_exact` (match po internal_sku/ean/asin),
   - `ean_match` (listing registry),
   - `embedding_match` (title similarity fallback).
3. Upsertuje wynik do `acc_taxonomy_prediction`.
4. (Opcjonalnie) auto-aplikuje rekordy powyżej progu confidence.

## Review queue
API:
- `POST /api/v1/inventory/taxonomy/refresh`
- `GET /api/v1/inventory/taxonomy/predictions`
- `POST /api/v1/inventory/taxonomy/predictions/{prediction_id}/review?action=approve|reject`

Rola:
- read: Analyst
- refresh: Ops
- approve/reject: Director

## Runtime integration
### Inventory (`manage_inventory.py`)
`_load_inventory_items()` ładuje lookup z `load_taxonomy_lookup(...)`.
`_build_item()` fallback:
- `brand = registry.brand -> product.brand -> taxonomy.brand`
- `category = registry.category_1 -> product.category -> taxonomy.category`
- `product_type = registry.category_2 -> product.subcategory -> taxonomy.product_type`

### Profit (`profit_engine.py`)
`get_product_profit_table()` ładuje lookup taxonomy i fallbackuje:
- `brand` oraz `category` przed filtracją i przed zwrotem response.

## Jobs
Dodano typ joba:
- `sync_taxonomy` w `mssql_store.py` (`ALLOWED_JOB_TYPES` + `run_job_type` branch).

Payload (opcjonalny):
- `limit`
- `min_auto_confidence`
- `auto_apply`

### Live progress
`sync_taxonomy` aktualizuje `acc_al_jobs` w trakcie runu:
- `processed/total`
- `remaining`
- `generated`
- `stage` (`building_candidates`, `predicting`, `auto_apply`, `done`)

### Nightly scheduler
Dodano harmonogram APScheduler w `app/scheduler.py`:
- job id: `sync-taxonomy-nightly`
- task: `_sync_taxonomy()`
- source: `trigger_source=scheduler`, `triggered_by=apscheduler`

Parametry z `app/core/config.py`:
- `TAXONOMY_SYNC_ENABLED`
- `TAXONOMY_SYNC_HOUR`
- `TAXONOMY_SYNC_MINUTE`
- `TAXONOMY_SYNC_LIMIT`
- `TAXONOMY_SYNC_AUTO_APPLY`
- `TAXONOMY_SYNC_MIN_CONFIDENCE`

## Notes
- `load_taxonomy_lookup()` używa tylko statusów: `auto_applied`, `applied`, `approved`.
- Schema bootstrap ma cache (`_SCHEMA_READY`), żeby nie wykonywać DDL checków przy każdym request.
